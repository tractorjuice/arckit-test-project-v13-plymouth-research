## 3. Context and Problem Statement

### 3.1 Problem Description

The project has a number of orphaned requirements related to data quality and governance that are not currently addressed by any architecture decision. These requirements are critical for ensuring the accuracy, completeness, and trustworthiness of the data presented in the analytics platform.

**Problem statement as a question**: How should the project implement a comprehensive data quality and governance framework to address the orphaned requirements and ensure the long-term quality and integrity of the data?

### 3.2 Why This Decision Is Needed

- **Business context**: The value proposition of the platform is built on providing high-quality, trustworthy data. Without a formal data quality framework, the data may become unreliable, eroding user trust and devaluing the platform.
- **Technical context**: The current data processing pipeline has some basic data quality checks, but it lacks a comprehensive framework for monitoring, measuring, and improving data quality over time.
- **Regulatory context**: While not a strict regulatory requirement, having a robust data governance framework is a best practice that can help demonstrate due diligence in case of any data-related issues.

## 4. Decision Drivers (Forces)

### 4.1 Technical Drivers

- **Data Quality**: The need to ensure the accuracy, completeness, timeliness, and uniqueness of the data.
- **Data Lineage**: The need to track the origin and transformation of data for debugging and audit purposes.
- **Data Governance**: The need to establish clear ownership, policies, and procedures for managing data assets.

### 4.2 Business Drivers

- **Trust**: The need to build and maintain user trust in the platform by providing accurate and reliable data.
- **Reputation**: The need to protect the reputation of Plymouth Research by ensuring the quality and integrity of its data.
- **Efficiency**: The need to improve the efficiency of data-related tasks by automating data quality checks and providing a clear data governance framework.

---

## 5. Considered Options

### Option 1: Implement a lightweight, custom data quality framework

**Description**: Develop a set of custom Python scripts and SQL queries to perform data profiling, data quality monitoring, and data lineage tracking.

#### Good (Pros)
- ✅ **Flexibility**: The framework can be tailored to the specific needs of the project.
- ✅ **Low cost**: No additional licensing costs.

#### Bad (Cons)
- ❌ **Development effort**: Requires significant development effort to build and maintain.
- ❌ **Limited features**: May lack advanced features found in off-the-shelf tools.

### Option 2: Use an open-source data quality tool

**Description**: Integrate an open-source data quality tool like Great Expectations or dbt with the existing data processing pipeline.

#### Good (Pros)
- ✅ **Rich features**: Provides a wide range of data quality features out of the box.
- ✅ **Active community**: Benefits from the support and contributions of a large open-source community.

#### Bad (Cons)
- ❌ **Learning curve**: Requires learning a new tool and its concepts.
- ❌ **Integration effort**: May require significant effort to integrate with the existing pipeline.

### Option 3: Do nothing

**Description**: Continue with the existing ad-hoc data quality checks.

#### Good (Pros)
- ✅ **No effort**: Requires no additional effort.

#### Bad (Cons)
- ❌ **High risk**: The risk of data quality issues will continue to grow as the platform evolves.
- ❌ **Technical debt**: Will create significant technical debt that will be difficult to address later.

---

## 6. Decision Outcome

### 6.1 Chosen Option

**"Option 2: Use an open-source data quality tool"**

### 6.2 Y-Statement (Structured Justification)

> **In the context of** the need to implement a comprehensive data quality and governance framework,
> **facing** the challenge of a large number of orphaned data quality requirements,
> **we decided for** integrating an open-source data quality tool like Great Expectations or dbt,
> **to achieve** a robust and feature-rich data quality framework with the support of an active open-source community,
> **accepting** the learning curve and integration effort required to adopt a new tool.

---

## 7. Consequences

### 7.1 Positive Consequences

- ✅ **Improved data quality**: The platform will have a robust framework for ensuring the quality and integrity of its data.
- ✅ **Increased trust**: Users will have more trust in the platform and its data.
- ✅ **Reduced risk**: The risk of data-related issues will be significantly reduced.

### 7.2 Negative Consequences (Accepted Trade-offs)

- ❌ **Increased complexity**: The data processing pipeline will become more complex.
- ❌ **Additional dependencies**: The project will have a new dependency on an open-source tool.

---

## 8. Validation & Compliance

### 8.1 How Will Implementation Be Verified?

- **Code review**: The integration of the data quality tool will be reviewed to ensure it meets the project's standards.
- **Testing strategy**: The data quality checks will be tested to ensure they are working correctly.
